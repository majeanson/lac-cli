User Authentication

Problem

Users need a secure way to log in and maintain sessions across the app.

Analysis

JWT-based auth with refresh tokens stored in httpOnly cookies. Stateless server, tokens expire in 15 minutes.

Decisions
  1. 1. Use JWT over server-side sessions 2026-01-10
    Stateless servers make horizontal scaling trivial and remove the need for a session store.
    Alternatives considered: Redis-backed sessions, Database sessions
Known Limitations
  • Token revocation requires waiting for expiry unless a denylist is added

Feature Schema

Problem

Every provenance system needs a canonical, versioned, machine-readable format for capturing why a feature exists. Without a shared schema, tooling cannot validate, index, or evolve feature records consistently.

Analysis

A Zod-based schema provides runtime validation and TypeScript type inference from a single source of truth. The featureKey pattern (domain-YYYY-NNN) encodes temporal context and supports multi-domain deployments. Optional fields are additive so the schema can evolve without breaking existing records. The schemaVersion field enables future migration tooling. As of iteration 3, 40 Vitest tests exercise all validation paths.

Decisions
  1. 1. Use Zod for runtime validation 2026-01-15
    Single schema definition drives both TypeScript types and runtime validation. No code duplication.
    Alternatives considered: JSON Schema only — no TS type inference, manual validation — fragile and verbose
  2. 2. featureKey format: domain-YYYY-NNN 2026-01-15
    Year in key makes keys self-documenting. Domain prefix enables multi-domain workspaces (proc-, goal-, adr-).
    Alternatives considered: UUIDs — opaque, not human-readable, sequential integers — no temporal context
  3. 3. schemaVersion as an explicit integer field 2026-02-01
    Enables forward migration when breaking changes are needed. Current value is 1.
    Alternatives considered: Implicit versioning via field presence — ambiguous under concurrent evolution
Implementation

Zod schema exported as FeatureSchema with validateFeature() and generateFeatureKey() utilities. The FEATURE_KEY_PATTERN regex accepts any lowercase domain prefix. A JSON Schema file (feature.schema.json) is co-located for IDE validation and autocompletion. Published as @life-as-code/feature-schema. Build output: ESM + .d.mts type declarations via tsdown.

Known Limitations
  • knownLimitations field is a flat string array — no structured impact/workaround fields yet
  • No schema migration tooling for breaking changes (schemaVersion field exists but no migrator)
  • annotations field not yet in schema — added ad-hoc by consuming packages

feat-2026-002 Active

lac CLI

Problem

Developers need a terminal interface for the entire feature provenance lifecycle — creating features, running quality checks, tracing ownership, and operating the local server — without leaving their workflow.

Analysis

A Commander.js-based CLI exposes 11 commands covering the full lifecycle. Commands are thin wrappers over shared lib/ utilities so they are individually testable. The serve command bridges the CLI and LSP worlds by spawning lac-lsp and opening the browser. As of iteration 3, 23 Vitest tests cover loadConfig, doctor, lint, and blame commands.

Decisions
  1. 1. Commander.js over yargs or meow 2026-01-20
    Commander has the cleanest subcommand model and TypeScript support without runtime type gymnastics.
  2. 2. lac serve spawns lac-lsp as a child process rather than embedding the HTTP server 2026-03-01
    Keeps packages independently deployable. lac-cli stays lightweight. lac-lsp can be used standalone.
Implementation

Commands: workspace (init/status), init, spawn, tag, archive, lint (--watch), blame, export (--site), hooks (install/uninstall), serve, doctor, search. Built with tsdown to ESM. Installed globally as lac binary. Reads lac.config.json from workspace root for default options and paths.

Known Limitations
  • lint --watch uses fs.watch which is unreliable on Linux (use chokidar in a future iteration)
  • No shell completion yet (bash/zsh/fish)
  • lac diff command not yet implemented

feat-2026-033 Frozen cli-export

lac export-all: batch all export formats to a folder

Problem

Running each export format individually requires 15+ separate CLI invocations. Users who want a full snapshot of their workspace — for sharing, archiving, or CI artifacts — have no single command to produce all outputs at once.

Analysis

The export-all command is a batch runner over the existing generator library. It scans features once from a given directory, then calls every multi-feature generator (14 HTML formats + prompt + site) and writes outputs to a single folder. The architecture is a thin registry pattern: an array of {name, file, generate} records drives the main loop, keeping the command extensible without code duplication. Non-applicable single-feature formats (postcard, markdown) and comparison formats (diff) are intentionally excluded. The site generator writes a directory tree rather than a single file, so it is handled separately with its own mkdir. A --skip flag accepts comma-separated format names so users can exclude slow or unwanted outputs. Errors per format are non-fatal — the loop continues and the summary reports failures.

Decisions
  1. 1. Registry array drives the loop rather than hardcoded if-chains
    Adding a new export format in the future requires one entry in the HTML_FORMATS array, not a new if-block. Mirrors the generator library's own open/closed structure.
    Alternatives considered: Individual if-blocks per format (copied from export.ts), Dynamic import via format name string
  2. 2. Scan features once, pass to all generators
    Avoids repeated filesystem walks. All generators accept the same ScannedFeature[] type, so one scan suffices for all 14+ formats.
    Alternatives considered: Re-scan per format (simpler but slow for large workspaces)
  3. 3. Exclude postcard, markdown, and diff from export-all
    postcard and markdown are single-feature formats that require a nearest feature.json context. diff requires a second workspace path. None are meaningful in a batch context.
    Alternatives considered: Include postcard using first frozen feature, Add --diff-target option
Implementation

HTML_FORMATS registry: array of {name, file, generate} where generate wraps each generator with the correct argument shape. ALL_FORMAT_NAMES: used for --skip validation. Action: resolves scanDir and outDir, validates skip names, scans features, mkdir outDir, iterates HTML_FORMATS (skipping or writing), handles prompt (buildReconstructionPrompt from export.ts, writes lac-prompt.md), handles site (generateSite to outDir/site/). Prints ✓/✗/skip per file. Summary line: N exported, M failed, K skipped.

Known Limitations
  • No --view or --tags support — export-all targets full snapshots, not filtered views
  • Site output is always written to <outDir>/site/ with no configurable subdirectory name
  • Generators that fail do not abort the run — partial output is possible without a clear error exit code
  • Does not support postcard or markdown (single-feature only) or diff (requires a second target)
Success Criteria

lac export-all with 3 features produces 16 files (14 html + lac-prompt.md + site/) in ./lac-exports/. --skip site,quiz produces 14 files. --skip unknownformat exits with an error message listing valid formats. Each generated file is non-empty.

feat-2026-003 Active

Language Server + HTTP API

Problem

Editors need real-time feature provenance overlaid on code (hover cards, code lens, diagnostics) without polling the filesystem on every keystroke. CI pipelines and browser tools need a stable HTTP API to query the same index.

Analysis

A single long-running process serves both transports. The LSP stdio transport connects to any LSP-capable editor. The HTTP server on :7474 provides a REST API for CI, browser extensions, and the webapp. A shared in-memory FeatureIndex (chokidar-backed) is the single source of truth for both transports. As of iteration 3, 10 Vitest tests cover HTTP routing, rate limiting, lineage, and oversized-body protection.

Decisions
  1. 1. Single process for both LSP and HTTP 2026-02-01
    One shared FeatureIndex means no sync needed between transports. Simpler to install and operate.
    Alternatives considered: separate processes — double the memory, sync complexity
  2. 2. Replace fs.watch with chokidar 2026-03-15
    fs.watch --recursive is unreliable on Linux/Docker. Chokidar provides consistent cross-platform behavior.
Implementation

LacLspServer orchestrates FeatureIndex + optional HttpServer + optional LSP connection. LSP capabilities: hover (with missing fields + completeness %), codeLens, definition, diagnostics (push), workspaceSymbols. HTTP endpoints: GET /health (uptime, indexedCount, version) /features /features/:key /features/:key/lineage /blame /lint /events (SSE), PUT /features/:key, DELETE /features/:key. Startup log: 'lac-lsp v0.1.0 started on port N'. File watching via chokidar.

Known Limitations
  • Only one workspace root supported — multi-root workspaces use first folder only
  • textDocument/references not yet implemented
  • textDocument/rename not yet implemented

feat-2026-004 Active

VS Code Extension (lac-lens)

Problem

Developers spend most of their time in their editor. Feature provenance needs to surface exactly where the code is — as a CodeLens badge, hover card, and status bar item — without requiring context switches to a browser or terminal.

Analysis

Two modes: Direct (reads feature.json via VS Code FS API, no external process) and LSP (delegates to lac-lsp for diagnostics and workspace symbols). Auto-detect: if lac-lsp is in PATH at activation, LSP mode is enabled automatically. This means zero configuration for the common case. As of iteration 3, hover provider escapes backticks in all string values to prevent broken markdown rendering.

Decisions
  1. 1. Two modes (direct + LSP) rather than LSP-only 2026-02-10
    Direct mode works offline, in remote environments, and without any global installs. LSP mode is opt-in for power users.
  2. 2. Auto-detect lac-lsp at activation via spawnSync 2026-03-15
    Zero-config experience. Most users who have lac-lsp installed want LSP mode. Explicit false opt-out is better than explicit true opt-in.
Implementation

CodeLens shows featureKey + status + completeness%. Hover shows problem, analysis, decisions, limitations with backtick escaping. Status bar shows active feature. Click-to-open feature.json. LSP mode adds diagnostics, go-to-definition, workspace symbol search. Bundled with esbuild to CJS for VS Code compatibility.

Known Limitations
  • Only first workspace folder used in multi-root workspaces
  • No Neovim/Helix/JetBrains support (LSP is standard but needs client config)
